# 节假日航线价格监控工具 v1 需求文档

## 1. 产品定位

面向个人节假日出游的机票价格监控与提醒工具。

v1 的目标不是做一个通用机票搜索平台，而是帮助用户在未来节假日出游前，持续监控从上海出发的候选航线价格，发现低价机会，并通过邮件及时提醒用户。

核心问题：

- 节假日想出去玩，但目的地可以灵活。
- 希望优先买到便宜机票。
- 目的地最好具备自然风景、历史人文景观，值得专程旅行。
- 直飞和中转都可以接受，但需要分开评估。
- 节假日和目的地应由系统优先推荐，用户可以直接采用推荐项，也可以手动输入。
- 前期个人使用，后期可能扩展为多人使用。

## 2. 目标用户

### v1 主要用户

个人用户，从上海出发，计划在节假日国内或国际旅行。

### 后续潜在用户

- 多个家庭成员共同关注出行计划。
- 小团队或朋友群共享候选目的地和价格提醒。
- 多用户系统，每个用户独立管理自己的航线、预算、提醒规则。

## 3. 用户偏好与约束

- 出发地：以上海为主。
- 出发机场：默认包含上海浦东国际机场和上海虹桥国际机场。
- 航线范围：同时支持国内航线与国际航线。
- 目的地策略：以便宜为主，但目的地需要具备旅游吸引力。
- 旅游吸引力偏好：
  - 自然风景，例如海岛、山川、湖泊、雪景、草原、温泉。
  - 历史人文，例如古城、博物馆、文化遗产、历史街区、宗教建筑。
- 航班策略：
  - 直飞作为独立策略记录。
  - 中转作为独立策略记录。
  - 两类策略不混在同一个价格判断里。
- 推荐策略：
  - 系统优先推荐适合监控的节假日。
  - 系统优先推荐适合从上海出发的目的地。
  - 用户可以选择系统推荐，也可以手动输入节假日、日期范围和目的地。
  - 默认体验应尽量减少用户输入，让用户从推荐列表中勾选和微调。
- 提醒方式：v1 首选邮件提醒。
- 使用范围：v1 面向个人使用，但数据结构和权限模型应预留多人扩展空间。

## 4. v1 范围

### v1 必须实现

1. 节假日管理
   - 系统内置中国常见节假日：
     - 元旦
     - 春节
     - 清明
     - 劳动节
     - 端午
     - 中秋
     - 国庆
   - 系统需要标记推荐监控的节假日。
   - 推荐节假日需要包含推荐理由，例如假期长度、适合长线或短线旅行、是否容易错峰。
   - 推荐节假日需要支持排序，默认优先展示距离当前时间较近、适合提前监控的假期。
   - 支持用户自定义假期名称和日期范围。
   - 支持为假期设置前后浮动天数，例如假期前后各 1 到 3 天。

2. 出发地与目的地管理
   - 默认出发城市为上海。
   - 默认出发机场包含 PVG 和 SHA。
   - 系统内置一批适合从上海出发的推荐目的地。
   - 推荐目的地需要覆盖国内、国际及港澳台。
   - 系统需要优先展示推荐目的地，用户可以从推荐列表中勾选。
   - 支持用户手动添加候选目的地。
   - 每个目的地需要记录：
     - 城市或地区名称
     - 国家或地区
     - 国内 / 国际
     - 主要机场
     - 目的地标签：自然风景、历史人文、海岛、亲子、滑雪、美食等
     - 推荐季节或适合节假日
     - 是否系统推荐
     - 推荐优先级
     - 推荐理由
     - 适合的节假日类型，例如短假、长假、春节、国庆、暑期

3. 航线监控任务
   - 用户可以创建监控任务。
   - 每个任务包含：
     - 出发城市或机场组
     - 候选目的地
     - 节假日或自定义日期范围
     - 单程 / 往返
     - 乘客人数
     - 舱位，v1 默认经济舱
     - 预算阈值
     - 是否允许中转
     - 直飞策略
     - 中转策略
   - 同一目的地的直飞和中转需要分别生成价格记录和提醒判断。

4. 系统推荐入口
   - 创建监控任务时，默认进入系统推荐流程。
   - 系统先推荐适合监控的节假日，再推荐目的地组合。
   - 用户可以选择：
     - 直接使用系统推荐。
     - 基于系统推荐修改日期、目的地、预算和航班策略。
     - 跳过推荐，完全手动输入。
   - 推荐结果需要说明推荐理由。
   - 推荐结果至少包含：
     - 推荐节假日
     - 推荐日期范围
     - 推荐目的地
     - 目的地标签
     - 适合直飞还是中转
     - 推荐理由
     - 默认预算建议
   - v1 的默认交互目标是让用户在少量选择后即可创建监控任务。

5. 价格采集
   - 定时查询每个监控任务的机票价格。
   - v1 建议每天查询 1 次。
   - 临近节假日可以提高频率，例如每天 2 到 4 次。
   - 每次采集至少记录：
     - 查询时间
     - 出发机场
     - 到达机场
     - 出发日期
     - 返程日期
     - 航司
     - 航班号或组合标识
     - 直飞 / 中转
     - 中转城市和中转次数
     - 总耗时
     - 价格
     - 币种
     - 是否含税费
     - 是否含托运行李
     - 来源平台
     - 购票链接或跳转信息

6. 价格历史
   - 保存每次查询结果。
   - 按航线、日期组合、策略类型统计历史最低价。
   - 支持查看：
     - 当前最低价
     - 历史最低价
     - 最近 7 天最低价变化
     - 最近 30 天最低价变化
     - 当前价格相对历史区间的位置

7. 低价判断
   - v1 至少支持三类提醒规则：
     - 价格低于用户设置的预算阈值。
     - 当前价格低于最近 30 天均价一定比例。
     - 当前价格刷新该任务历史最低价。
   - 直飞和中转分别判断，分别提醒。
   - 提醒内容需要说明为什么触发，而不只是发送价格。

8. 邮件提醒
   - v1 使用邮件作为主要提醒方式。
   - 邮件内容包含：
     - 节假日名称
     - 航线
     - 出发 / 返程日期
     - 当前价格
     - 历史参考
     - 直飞或中转策略
     - 航司与耗时
     - 触发原因
     - 购票链接或来源说明
   - 同一任务短时间内避免重复提醒。
   - 支持简单的提醒冷却时间，例如 12 小时或 24 小时。

9. 目的地推荐基础评分
   - v1 需要给目的地一个简单的旅行价值评分。
   - 评分不需要复杂模型，但需要可解释。
   - 建议维度：
     - 价格吸引力
     - 自然风景匹配度
     - 历史人文匹配度
     - 航程便利度
     - 签证或入境便利度
     - 节假日适配度
     - 从上海出发的航线丰富度
   - v1 的推荐排序以价格为第一优先级，旅行价值为第二优先级。
   - 系统推荐目的地时，需要结合节假日类型：
     - 短假优先推荐飞行时间短、直飞多的目的地。
     - 长假可以推荐更远距离或中转性价比高的目的地。
     - 春节和冬季假期可提高温暖海岛、雪景、温泉目的地权重。
     - 国庆和五一可提高自然风景、历史人文、城市漫游目的地权重。

10. 基础管理界面或命令入口
   - v1 可以先做简单管理界面或配置文件方式。
   - 必须支持：
     - 查看系统推荐节假日
     - 查看系统推荐目的地
     - 新增监控任务
     - 暂停 / 恢复监控任务
     - 查看任务列表
     - 查看某个任务的价格历史
     - 查看最近触发的提醒

## 5. v1 不做的事情

- 不直接完成支付或出票。
- 不做复杂会员体系。
- 不做社交分享。
- 不做完整 OTA 平台。
- 不保证价格预测准确性。
- 不在 v1 做微信、短信、App 推送。
- 不在 v1 做多人协作界面，但数据模型预留用户字段。
- 不在 v1 做全自动“帮我买票”，只做提醒和决策辅助。

## 6. 核心使用流程

### 流程一：创建节假日监控

1. 系统展示推荐节假日，例如即将到来的国庆、春节、五一。
2. 用户选择一个系统推荐节假日，或手动输入自定义日期范围。
3. 系统生成可出行日期范围和建议浮动范围。
4. 系统根据节假日、上海出发、价格优先、自然风景和历史人文偏好推荐目的地。
5. 用户从推荐目的地中勾选，或手动添加目的地。
6. 系统给出默认预算建议，用户可以修改预算。
7. 用户选择是否同时监控直飞和中转。
8. 系统创建监控任务。

### 流程二：使用系统推荐快速创建监控

1. 用户进入推荐页。
2. 系统展示推荐卡片，例如“国庆 7 天，日本关西历史人文”“春节 6 天，东南亚海岛避寒”“五一 4 天，国内山水短途”。
3. 用户选择一个推荐卡片。
4. 系统自动填充节假日、日期范围、目的地组、预算建议、直飞 / 中转策略。
5. 用户确认或微调。
6. 系统创建监控任务。

### 流程三：系统定时采集价格

1. 系统读取所有启用中的监控任务。
2. 根据日期范围生成多个出发 / 返程组合。
3. 分别查询直飞方案和中转方案。
4. 保存查询结果。
5. 更新历史最低价和近期均价。
6. 判断是否触发提醒。

### 流程四：邮件提醒

1. 系统发现低价或历史低价。
2. 检查该任务是否处于提醒冷却期。
3. 如果允许提醒，生成邮件。
4. 邮件发送给用户。
5. 系统记录提醒日志。

### 流程五：用户查看价格历史

1. 用户打开任务详情。
2. 查看当前最低价。
3. 对比直飞与中转。
4. 查看价格曲线。
5. 根据价格、耗时、目的地吸引力决定是否购票。

## 8. 初始目的地建议

v1 可以先建立一批适合从上海出发的候选目的地。

### 国内

- 北京：历史人文强，航班多。
- 西安：历史人文强，适合短假期。
- 成都：人文、美食、周边自然风景。
- 重庆：城市景观、美食、周边自然风景。
- 昆明：气候稳定，自然风景延展性强。
- 丽江 / 大理：自然风景和人文兼具。
- 桂林：自然风景强。
- 张家界：自然风景强。
- 厦门：海滨、人文、短途友好。
- 三亚 / 海口：海岛度假。
- 哈尔滨：冬季和春节适配度高。
- 乌鲁木齐：自然风景强，但航程较长。

### 国际及港澳台

- 香港：城市、人文、转机便利。
- 澳门：短途、人文、休闲。
- 台北：人文、美食、自然延展。
- 东京：城市、人文、周边自然景观。
- 大阪 / 京都：历史人文强。
- 札幌：冬季自然风景强。
- 首尔：短途、人文、购物、美食。
- 济州：自然风景、短途。
- 曼谷：价格常有优势，城市和周边文化景点。
- 清迈：自然与人文兼具。
- 普吉 / 甲米：海岛自然风景。
- 新加坡：城市、人文、亲子友好。
- 吉隆坡：价格常有优势，中转和城市旅行。
- 槟城：历史人文、美食。
- 河内 / 胡志明市：人文、美食。
- 岘港：海滨和周边历史文化。
- 巴厘岛：自然风景和度假。

## 10. 价格与推荐排序

v1 推荐列表建议按以下逻辑排序：

1. 是否来自系统推荐。
2. 是否触发低价提醒。
3. 当前价格是否低于预算。
4. 当前价格相对历史价格是否足够低。
5. 直飞优先或中转优先，由用户配置决定。
6. 目的地旅行价值评分。
7. 总飞行耗时。
8. 行李和税费是否包含。

默认排序：低价优先，其次考虑目的地旅行价值。

系统推荐排序：节假日匹配度优先，其次考虑价格潜力、旅行价值、航线便利度。

## 11. 直飞与中转策略

### 直飞策略

适合：

- 短假期。
- 家庭出行。
- 时间价值高于价格差的场景。

记录重点：

- 价格
- 航司
- 起降时间
- 总耗时
- 是否早去晚回

### 中转策略

适合：

- 长假期。
- 国际航线。
- 价格差明显的场景。

记录重点：

- 价格
- 中转城市
- 中转次数
- 中转时长
- 总耗时
- 是否需要过境签
- 是否分段购票

v1 需要避免将直飞价格和中转价格混在同一个历史最低价中，否则会影响判断。

## 12. 邮件提醒样例

邮件标题：

```text
[机票提醒] 国庆 上海 -> 大阪 中转价格刷新低价：¥1,680
```

邮件正文应包含：

```text
节假日：国庆
航线：上海 -> 大阪
日期：2026-10-01 出发，2026-10-07 返回
策略：中转
当前价格：¥1,680
历史最低：¥1,680
最近 30 天均价：¥2,350
触发原因：当前价格刷新历史最低，且低于 30 天均价 28%
航司：示例航空
耗时：7 小时 40 分钟
中转：首尔，1 次中转
```

## 13. 成功指标

v1 是否成功，可以用以下指标判断：

- 用户可以在 5 分钟内创建一个节假日监控任务。
- 用户可以通过系统推荐在 2 分钟内创建一个监控任务。
- 系统可以稳定保存每次价格查询结果。
- 系统可以区分直飞和中转的价格趋势。
- 系统可以在低价出现时发送邮件。
- 用户收到邮件后能快速判断是否值得购买。
- 至少支持 20 个候选目的地。
- 至少支持 7 个系统推荐节假日，覆盖元旦、春节、清明、劳动节、端午、中秋和国庆。
- 至少支持 10 个系统推荐目的地组合。
- 至少支持 3 个节假日监控任务同时运行。

## 14. 待确认问题

- 使用哪个机票价格数据源。
- 是否优先接入官方 API，还是先使用手动导入 / 半自动查询方式。
- 价格采集是否需要代理、登录、验证码处理。
- 国际航线是否需要记录签证信息。
- 预算按单人票价还是总人数总价提醒。
- 是否需要区分成人、儿童、婴儿票。
- 邮件服务使用 SMTP 还是第三方邮件 API。
- 第一版是否需要 Web UI，还是先以配置文件加邮件提醒启动。
- 系统推荐的默认预算如何生成，是人工维护还是基于历史价格统计。
- 推荐目的地组合由人工配置，还是由标签和评分规则自动生成。
